/goal Build a polished, runnable v0.1 of Stream Reliability Lab in this repository.

Read AGENTS.md, PLAN.md, PROGRESS.md, and DECISIONS.md first. Treat PLAN.md's acceptance criteria as the product contract and AGENTS.md as persistent working rules.

Work autonomously toward the complete first usable product rather than stopping after scaffolding. Begin by delegating read-only analysis in parallel to `reliability_architect` and `test_engineer`; wait for their summaries, make the final architecture decisions in the main thread, and record meaningful choices in DECISIONS.md. The main thread should own code changes so parallel agents do not conflict in the working tree.

Build in checkpoints, keeping the repository runnable after each checkpoint:

1. project/tooling skeleton and health check;
2. deterministic versioned event model and simulator;
3. WebSocket ingestion, structured acknowledgments, and DuckDB persistence-before-ack;
4. idempotent processing, overlay dispatch, browser rendering, and render acknowledgments;
5. duplicate, invalid-payload, delayed/out-of-order, reconnect, and reconnect-burst scenarios;
6. tested analytics endpoints and Streamlit dashboard;
7. Playwright end-to-end verification, Docker Compose, GitHub Actions, and portfolio README;
8. stabilization and cleanup without adding scope.

Use the simplest local architecture that satisfies the contract. You may make reasonable implementation decisions without asking. Do not add real platform integrations, credentials, cloud services, Kafka, Kubernetes, React, authentication, payments, or AI features. Do not publish, deploy, push, or open a pull request.

Continuously update PROGRESS.md with the active checkpoint, verified evidence, remaining work, blockers, and exact validation commands. Add or update tests with each behavior. Do not weaken tests to get green results.

The primary demonstration must let a reviewer start the application, open the overlay, run a deterministic roughly 500-event `reconnect_burst` scenario with a fixed seed, see synthetic effects render, and inspect lifecycle counts, duplicate suppression, invalid deliveries, latency percentiles, and reconnect evidence in the dashboard.

Before stopping:

- run the full documented unit, integration, Playwright, lint, formatting, and type-check suite;
- run the primary demo from a clean application state;
- delegate final read-only reviews in parallel to `skeptical_reviewer` and `simplicity_critic`;
- validate their findings and fix all verified critical/high defects plus material test gaps;
- rerun the complete verification suite;
- verify the README commands against the real implementation;
- leave no known broken or half-migrated path;
- report exact commands/results and remaining limitations honestly;
- do not invent benchmark or reliability numbers.

Stop when the complete v0.1 acceptance criteria in PLAN.md are met, the documented demo works, all required checks pass, and no verified critical/high review finding remains. If a noncritical improvement remains, document it instead of expanding the scope indefinitely.
